OBT Exception Monitor 사용 가이드

OBT Exception Monitor User Guide

OBT 적재 데이터를 기준으로 구간, 화주, 영업사원별 조치 우선순위를 읽는 상세 설명서

Detailed guide for route, customer, and sales-owner exception actions based on OBT loaded booking data

대시보드 Dashboard

이 화면이 답해야 하는 질문

이 모니터는 “잘 되고 있는 곳”이 아니라 “한정된 영업 시간을 어디에 써야 하는지”를 찾기 위한 화면입니다. 국가, 선적지, 도착국가, 도착포트, 영업사원을 선택하면 OBT 기준의 문제 구간과 담당자별 조치 후보가 먼저 표시됩니다.

접속 시 기본 대응 시점은 3주뒤(W+3)입니다. 화면의 구간별 특이사항 부제에는 대상 날짜와 WW 주차가 함께 표시됩니다.

What This Dashboard Answers

This monitor is designed to show where limited sales time should be spent, not to celebrate healthy lanes. Choose origin, POL, destination country, destination port, and sales owner to surface OBT-based route exceptions and owner actions.

The default horizon on entry is W+3. The route-exception subtitle displays the target date and the corresponding WW week number.

먼저 볼 것Start Here 구간별 특이사항의 P1/P2, 판단, 권장 조치를 먼저 확인합니다. Review P1/P2, judgment, and recommended action in Route Exceptions first.
조치 기준Action Basis BSA/물량이 큰 구간, 도착포트별 p75 이상 영향 구간을 우선 노출합니다. Focus routes are selected by BSA/volume and destination-port p75 impact thresholds.
담당 확인Owner Check 영업사원별 카드에서 상태, Gap, 회복/재확보 후보를 확인합니다. Use sales-owner cards to see status, action gap, and recovery/win-back candidates.

2. 대응 시점과 주차 계산

2. Horizon And Week Logic

대응 필터는 영업 대응 시점을 의미합니다. 데이터 기준일이 속한 주의 시작일을 기준으로 차주, 차차주, 3주뒤를 계산합니다.

필터의미권장 판단
차주(W+1)현재 주 시작일 + 7일출항 임박. BSA Gap, 미소석, 이탈 화주를 즉시 대응합니다.
차차주(W+2)현재 주 시작일 + 14일회복 시간이 제한적입니다. 감소 화주 회복과 대체 물량을 병행합니다.
3주뒤(W+3)현재 주 시작일 + 21일기본값입니다. 선행부킹 약화, 화주수 감소, BSA 목표 적정성을 먼저 봅니다.
차주~3주뒤W+1, W+2, W+3 합산전체 대응 규모를 볼 때 사용합니다. 주차별 액션은 별도로 나누는 것이 좋습니다.
대상 주차 = 데이터 기준일이 속한 주 시작일 + (대응 offset * 7일)
WW = 월별 주차 테이블 기준 누적 주차. 화면에는 예: 05/17 (WW20) 형식으로 표시됩니다.
BSA 매칭 = 선택 주차의 YYYYMM + WW가 모두 일치하는 BSA만 합산
차주~3주뒤 = W+1, W+2, W+3 각 주차 BSA 합
월 전체 = 선택 월의 전체 BSA 합

3주뒤를 선택하면 비교 기준도 같은 “3주뒤 상태”에 맞춰집니다. 예를 들어 금주가 WW17이면 W+3는 WW20이고, 전월 비교는 전월의 동일 슬롯 주차를 W+3 리드타임 기준으로 보정합니다.

The Horizon filter represents the sales response timing. W+1, W+2, and W+3 are calculated from the start date of the week containing the data date.

FilterMeaningRecommended Reading
W+1Current week start + 7 daysNear sailing. Act immediately on BSA gap, underfill, and lost customers.
W+2Current week start + 14 daysRecovery time is limited. Run customer recovery and substitute-volume actions together.
W+3Current week start + 21 daysDefault. Check weak advance bookings, fewer active customers, and BSA target realism.
W+1 to W+3Combined W+1, W+2, W+3Use for total action size. Operational actions should still be split by horizon.
Target week = week start of data date + (horizon offset * 7 days)
WW = cumulative week number from the monthly week table. The screen displays it as 05/17 (WW20).
BSA match = sum only BSA rows where selected YYYYMM and WW both match
W+1 to W+3 = sum the BSA of each selected week
Month view = sum the selected month BSA

When W+3 is selected, the comparison baseline is also aligned to the W+3 timing. For example, if the current week is WW17, W+3 is WW20, and previous-month comparison uses the same week slot adjusted to the W+3 lead-time stage.

3. 비교 기준과 리드타임 보정

3. Baseline And Lead-Time Alignment

주차별 대응은 “동일 최종 주차”가 아니라 “동일 리드타임 상태”로 비교해야 합니다. 3주뒤 물량을 이미 마감에 가까운 과거 최종 TEU와 비교하면 대부분 부족으로 보이기 때문입니다.

비교기준보정 방식
전월전월 동일 슬롯 주차W+3는 w3_fst, W+2/W+1은 fst와 w3_fst의 성숙도 보정
전주차선택 주차의 직전 주차동일 offset에 맞춰 비교
최근 3개월 평균최근 3개월 동일 슬롯 주차3개월 값을 합산 후 3으로 나누고 리드타임 보정
W+3 기준 BKG = baseline.w3_fst
W+2 기준 BKG = w3_fst + (fst - w3_fst) * 45%
W+1 기준 BKG = w3_fst + (fst - w3_fst) * 75%

이 보정은 데이터 제약을 고려한 운영 기준입니다. 향후 WOS-2, WOS-1 원천 컬럼이 추가되면 보정값이 아니라 실제 시점별 TEU로 대체하는 것이 가장 좋습니다.

Horizon comparison should use the same lead-time stage, not final historical TEU. Comparing W+3 volume against near-final historical TEU would overstate shortages.

CompareBaselineAdjustment
Previous monthSame week slot in previous monthW+3 uses w3_fst; W+2/W+1 use maturity adjustment between fst and w3_fst
Previous weekWeek immediately before the selected weekAligned to the same horizon offset
3-month averageSame week slot across last three monthsSum, divide by three, then align to lead-time maturity
W+3 baseline BKG = baseline.w3_fst
W+2 baseline BKG = w3_fst + (fst - w3_fst) * 45%
W+1 baseline BKG = w3_fst + (fst - w3_fst) * 75%

This is an operational adjustment under current source limitations. If WOS-2 and WOS-1 source fields become available, the dashboard should replace maturity estimates with actual time-stage TEU.

구간 테이블 지표

항목목적계산 수식해석
기준 BKG현재 BKG가 정상 수준인지 비교전월/전주/3개월 평균 BKG를 선택 W+ 시점으로 보정현재 BKG가 기준보다 낮으면 감소 신호입니다.

증감률기준 대비 감소 강도 확인(currentTeu - baseTeu) / baseTeu기준이 있는 구간에서 감소율을 판단합니다.

리드타임 트렌드와 속도 계산

이 영역은 “현재 W+ 시점에서 정상적인 속도로 물량이 들어오고 있는가”를 봅니다. 최종 물량이 아니라 같은 리드타임 단계의 성숙도를 비교합니다.

8. Sales Owner Action Panel

영업사원별 카드는 단순 건수보다 “담당자가 지금 어떤 상태인지”를 보여주기 위한 영역입니다.

항목계산/의미활용

영향 TEU담당 화주의 감소/이탈 영향 TEU 합계실제 물량 영향이 큰 담당자를 우선 봅니다.

지연구간리드타임 트렌드 또는 최근 부킹속도 기준보다 느린 담당 구간 수0이면 해당 담당자는 구간 지연보다 화주 감소/이탈, 3W 약화, BSA 목표 Gap 때문에 표시된 것입니다.

대표 구간영향 TEU가 가장 큰 구간담당자와 대화할 때 첫 확인 구간입니다.

메시지는 상황별로 다르게 나옵니다. 예를 들어 BSA Gap이 크면 “회복 후보와 대체 물량 분리”, 이탈 화주가 많으면 “재확보 가능성과 경쟁사 전환 사유 확인”, 감소 화주가 많으면 “회복 가능 잔여 물량 확인”으로 표시됩니다.

9. 화주 단위 Action List

9. Customer Action List

구간 단위 판단은 “어디가 문제인지”를 보여주고, 화주 단위 Action List는 “누구에게 연락해야 하는지”를 보여줍니다.

  1. P1 화주부터 확인합니다.
  2. 구간의 판단이 BSA 목표 Gap인지, 리드타임 부족인지 먼저 봅니다.
  3. 구간 열의 구간BSA는 화주별 배정이 아니라 해당 Route 전체의 BSA이며, 소석은 Route 전체 BKG/BSA, 구간Gap은 Route 전체 현재 BKG를 뺀 부족분입니다.
  4. 현재 BKG가 0이면 재확보 접근, 일부 남아 있으면 회복 접근을 합니다.
  5. 3W TEU 감소는 차주 이후 선적 계획과 조기 부킹 가능성을 확인합니다.
  6. Late 의존 또는 3W 취소위험은 부킹 품질과 취소 사유를 확인합니다.

Route-level judgment shows where the problem is; the Customer Action List shows whom to contact.

  1. Start with P1 customers.
  2. Check whether the route judgment is BSA target gap or lead-time shortage.
  3. Route BSA is the route-level BSA, not a customer-specific allocation. Util is route BKG/BSA, and Route Gap is BSA minus the route's current BKG.
  4. If current BKG is zero, use a win-back approach; if some BKG remains, use recovery.
  5. For 3W TEU decline, confirm future shipment plan and early-booking feasibility.
  6. For Late dependency or 3W cancellation risk, check booking quality and cancellation reasons.

10. 권장 사용 흐름

10. Recommended Workflow

  1. 대응 시점 분리: 기본 3주뒤에서 시작한 뒤 차차주, 차주 순서로 좁힙니다.
  2. 목적지 필터: 도착국가와 도착포트로 실제 관리 단위를 맞춥니다.
  3. 판단 확인: 리드타임 부족, BSA 목표 Gap, 화주 기반 감소 중 무엇인지 구분합니다.
  4. 영업사원 배정: 담당자 카드에서 상태와 관리 Gap을 보고 우선순위를 정합니다.
  5. 화주 접촉: Action List의 P1/P2 화주를 회복, 재확보, 선행전환, 방어로 나누어 처리합니다.
  6. BSA 재판단: 회복 후보로도 Gap을 채우기 어렵다면 대체 물량 또는 BSA 조정을 검토합니다.
  1. Split horizon: start from default W+3, then review W+2 and W+1.
  2. Filter destination: use destination country and port to match the real management unit.
  3. Read judgment: distinguish lead-time shortage, BSA target gap, and customer-base decline.
  4. Assign owners: prioritize sales owners by status and action gap.
  5. Contact customers: split P1/P2 customers into recovery, win-back, advance conversion, and protection.
  6. Recheck BSA: if recoverable candidates cannot fill the gap, review substitute volume or BSA adjustment.

11. 해석 시 주의사항과 개선 후보

11. Caveats And Future Improvements

  • 현재 W+2/W+1 비교는 실제 WOS-2/WOS-1 값이 아니라 보정값입니다.
  • 일별 속도는 history.json에 누적된 스냅샷이 많을수록 안정적으로 계산됩니다.
  • 작은 포트는 p75 기준이 낮게 잡힐 수 있어 기본 최소 기준을 함께 사용합니다.
  • BSA 목표 Gap은 영업 부족만 의미하지 않습니다. 목표 선복 자체가 과도한지 함께 봐야 합니다.
  • 향후 개선 데이터: WOS-2/WOS-1 TEU, 운임/CM1, 장비 제약, 선박별 BSA 변경 이력, 경쟁사 전환 사유 코드.
  • Current W+2/W+1 comparison uses maturity adjustment, not actual WOS-2/WOS-1 values.
  • Daily pace becomes more stable as history.json accumulates more snapshots.
  • Small ports can produce low p75 thresholds, so minimum thresholds are also applied.
  • BSA target gap is not always a sales failure. Review whether the target capacity itself is realistic.
  • Future data improvements: WOS-2/WOS-1 TEU, rate/CM1, equipment constraints, vessel-level BSA history, competitor-switch reason codes.
